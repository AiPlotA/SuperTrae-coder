# Office Hours 技能（产品 reframing）

> 基于 gstack `/office-hours` Skill 改造
> 目的：在写任何 spec 之前,把模糊产品意图 reframe 成具体的产品问题
> 强制级别：P1 - 任何"新产品 / 新模块 / 重塑"类需求必须先走

---

## 触发条件

满足以下**任一**时自动加载：

- 用户说"我想做 X"、"帮我想个 idea"、"评估下我的产品"、"重塑 XX"
- 新项目立项 / 全新模块（greenfield）
- 用户对"要做什么"描述模糊（"做个类似 X 的东西"）
- 启动 brainstorming 之后、写 spec 之前

**前置依赖**：应已走完 `skill/brainstorming`（5 问 + 3 方案 + HARD-GATE）。

---

## 5 强制问题

每个问题**必须**与用户走完，禁止跳过：

### 问题 1：目标用户是谁？

> 不要接受"所有人" / "开发者" / "企业"这种宽泛描述。追问到具体画像：
> - 哪个角色？（例：后端工程师 / 30-50 人创业公司的 CTO / 大学生）
> - 在什么场景下会用？（例：凌晨 3 点紧急修 bug 时 / 给非技术创始人解释技术债时）
> - 已用什么替代方案？（例：Notion 笔记 / Excel / 心理记忆）

### 问题 2：核心痛点是什么？

> 不要接受"提高效率" / "更方便"这种宽泛描述。追问到具体可测量的痛：
> - 痛点多频繁？（例：每天 3 次 / 每周 1 次）
> - 痛到什么程度？（例：损失 X 元 / 浪费 Y 小时 / 错过 Z 机会）
> - 痛的最深一层是什么？（5 Whys）

### 问题 3：现有替代方案是什么？它们为什么不够好？

> 不要接受"市面上没有" / "都太复杂"。追问到具体对比：
> - 列 3 个最常用的替代品（含人工流程）
> - 每个替代品最让用户失望的 1 点
> - 我们的差异化点具体是什么？

### 问题 4：成功标准是什么？（可度量）

> 不要接受"好用"、"用户喜欢"这种主观描述。追问到可度量：
> - 北极星指标是什么？（例：DAU / 周活 / 转化率 / NPS）
> - 3 个月内要达到的具体数字？
> - 怎么知道达到了？（数据埋点 / 用户访谈 / A/B 测试）

### 问题 5：边界（不做什么）是什么？

> 不要接受"先做着看"。追问到明确边界：
> - 不做的 3 件事是什么？（例：不做移动端 / 不做 i18n / 不做企业 SSO）
> - 不做的话会失去什么用户？值得吗？
> - 边界会随什么变化而调整？

---

## 评分模板（产品 reframing 完整度，0-10 分）

| 维度 | 0 分（无内容）| 5 分（有方向但模糊）| 10 分（清晰可执行）|
|------|-------------|------------------|------------------|
| 目标用户 | "所有人" | 角色明确但场景模糊 | 角色 + 场景 + 已用替代品全清 |
| 痛点 | "提高效率" | 痛点存在但不可测量 | 频率 + 程度 + 5 Whys 全清 |
| 替代方案 | 没调研 | 列出 2-3 个,缺差异化 | 3 个替代品 + 各 1 个最差 + 我们 1 个差异点 |
| 成功标准 | "好用" | 有指标但无数字 | 北极星 + 3 个月数字 + 验证方法 |
| 边界 | "先做着看" | 模糊的不做项 | 3 件明确不做 + 失去用户评估 |

**总分** = 5 维度平均分

- **0-3 分**：产品想法还没 ready,继续聊
- **4-6 分**：可以进 brainstorming 的 3 方案
- **7-10 分**：可以直接进 spec-driven-development

## Fix 输出格式（reframing 不完整时）

```markdown
## Reframing 缺口 + 修复方向

### 缺口 1：[维度名] 当前 3/10
- 现状：[用户说的内容]
- 缺什么：[缺的具体元素]
- 修复问题：[给用户追问的 1 个具体问题]

### 缺口 2：[维度名] 当前 5/10
- 现状：[用户说的内容]
- 缺什么：[缺的具体元素]
- 修复问题：[给用户追问的 1 个具体问题]

## 下一步

- 5 维度全过 7 分后,自动加载 skill/spec-driven-development
- 3 个缺口修复后,重跑本 skill 评分
```

## 反模式（任何一条触发即视为违规）

- ❌ 接受"目标用户是所有人"而不追问
- ❌ 接受"成功标准是用户喜欢"而不追问可度量指标
- ❌ 跳过"边界"问题直接进 spec
- ❌ 用 1 个 5 分钟就跑完所有问题（实际需要 15-30 分钟）
- ❌ 把 5 维度的答案写在"实现细节"里（例：用户问"用什么数据库"这不是 reframing 该回答的）

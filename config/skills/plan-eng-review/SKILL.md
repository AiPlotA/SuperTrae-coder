## 4 维度 0-10 评分模板

### 维度 1：数据流（Data Flow）—— 数据从进入到产出的路径清晰吗？

| 分 | 长什么样 |
|----|---------|
| 0 | 无数据流图 |
| 3 | 有 happy path 数据流 |
| 5 | 完整 happy + 2 个异常路径 + 数据落库点 |
| 7 | 含数据一致性 / 事务边界 / 幂等性 |
| **10** | **任意 1 个数据点能 30 秒内追到"从哪来、存哪、谁改过、给谁用"** |

### 维度 2：模块边界（Module Boundary）—— 模块间接口契约明确吗？

| 分 | 长什么样 |
|----|---------|
| 0 | 无模块划分 / 全部耦合 |
| 3 | 有目录分层但无接口契约 |
| 5 | 有 module + 公开 API 列表 |
| 7 | 有接口契约 + mock 测试 + 依赖倒置 |
| **10** | **任意 1 个模块可被 mock 完整替代,其他模块 0 改动** |

### 维度 3：边界情况（Edge Cases）—— 异常 / 边界 / 并发都覆盖了吗？

| 分 | 长什么样 |
|----|---------|
| 0 | 只考虑 happy path |
| 3 | 列出 2-3 个边界 |
| 5 | 完整边界列表（空值 / 超长 / 并发 / 乱序 / 超时）|
| 7 | 每个边界有"用户可见行为" + "测试用例" |
| **10** | **穷举所有失败模式,任意 1 个失败都有 graceful 降级 + 用户可理解错误** |

### 维度 4：测试策略（Test Strategy）—— 覆盖率 + 隔离 + 回归

| 分 | 长什么样 |
|----|---------|
| 0 | 无测试 |
| 3 | 单元测试覆盖率 ≥50% |
| 5 | 单元 ≥80% + 关键路径有集成测试 |
| 7 | 单元 ≥80% + 集成 + E2E + 回归测试 |
| **10** | **测试金字塔完整 + mutation testing ≥70% + 性能基线 + 契约测试** |

## 3 强制问题

### 问题 1：5 个最容易出 bug 的边界情况是什么？

> 不要接受"输入校验"。追问到具体场景：
> - 用户传 null / 空数组 / 100MB 字符串?
> - 网络中断时正在提交?
> - 两个用户同时改同一行?
> - 时区 / 编码问题?
> - 数据库连接池耗尽?

### 问题 2：模块间的接口契约是什么？

> 不要接受"调用其他模块的 API"。追问到：
> - 输入 / 输出的 schema（不仅是类型,还有不变量）
> - 失败时的行为（重试 / 降级 / 抛错）
> - 版本兼容策略

### 问题 3：覆盖率目标 + 测量方法是什么？

> 不要接受"尽量多写测试"。追问到：
> - 单元测试覆盖率目标（默认 ≥80%）
> - 关键路径的 E2E 覆盖
> - mutation testing 验证测试有效性
> - CI 集成 + 覆盖率不达标时 fail build

## 反模式

- ❌ 数据流图用 Visio 画了但不更新（文档与代码 drift）
- ❌ 模块边界在 PR review 时才讨论（应 spec 阶段就锁定）
- ❌ "我们会处理边界"（不列举具体边界 = 不处理）
- ❌ 测试覆盖率 100% 但全是 mock 不验证行为

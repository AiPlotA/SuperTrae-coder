# 系统调试技能 v4.0（Systematic Debugging）

> 基于 superpowers systematic-debugging Skill + 工程哲学 v1
> 目的：找到 Bug 根因，而非症状修补
> 强制级别：P0 - 任何 Bug 修复必须遵守
> v4.0 强化：与 rules/core.md 9-13 条铁律联动，3 个新强制步骤

---

## 触发条件

- 用户报告 Bug
- 测试失败需要分析
- 出现未预期的行为
- 性能问题排查
- 错误信息（`TypeError` / `ReferenceError` / `NullPointer` / `panic` 等）
- 用户问"为什么不工作" / "为什么报错" / "为什么失败"

---

## 核心铁律（与 rules/core.md 9-13 条联动）

**9. 零编辑红线**：bug 报告必须触发代码修改
**10. 复现优先红线**：bug 修复前必须先主动制造 RED
**11. 4 阶段红线**：必须走 复现→定位→根因→修复+回归测试
**12. 修测试不修实现**：发现测试 bug 修测试
**13. 禁止删除验证设施**：不删 vitest.config.ts 等

**禁止"猜猜看"式调试（禁止随机修改代码看是否解决问题）**

**调试失败的最高代价是：浪费时间在错误的方向上**

## 4 阶段方法论（v4.0 强化版）

### 🆕 阶段 0：决策树（用户报错 vs disk 状态）

> v4.0 新增。**首先**判断用户报告与 disk 上的代码状态是否一致。

#### 决策流程

```
用户报告 bug
  │
  ├─ A. disk 上代码与用户报错一致（如用户说"TypeError on line 5"，disk line 5 确实有 bug）
  │   → 直接进入阶段 1
  │
  ├─ B. disk 上代码与用户报错不一致（用户说"X 错"，但 disk 上是 Y）
  │   │
  │   ├─ B1. 用户报错本身合理（按报错推论应该存在某段代码）
  │   │   → 🆕 主动制造 RED：在 disk 上**写入能复现用户报错**的测试代码
  │   │   → 跑测试 → 看到 RED
  │   │   → 进入阶段 1
  │   │
  │   └─ B2. 用户报错不合理（推论不出任何合理代码）
  │       → 与用户 HARD-GATE 确认："你提供的报错与当前代码不符，能否提供：
  │          1) 完整的代码截图，2) 完整的测试输出，3) 复现步骤？"
  │       → 确认后再继续
  │
  └─ C. 无法判断（如用户描述模糊）
      → 进入阶段 1 同时主动制造 RED（最保守路径）
```

#### 反模式（v4.0 严禁）

- ❌ **拒绝**：看到 disk 与用户报错不符就告诉用户"disk 上没问题，请检查环境"
- ❌ **搪塞**：声称"测试已通过"但实际上测试根本没跑起来（缺 config）
- ❌ **零编辑**：收到 bug 报告 12+ 次循环思考后不改一行代码
- ✅ **正解**：主动制造 RED → 跑测试 → 看到真实失败 → 进入阶段 1

---

### 阶段 1：复现（Reproduce）

**目标**：在受控条件下稳定重现 Bug

#### 执行步骤

1. **精确描述现象**
   - 输入是什么？
   - 实际输出是什么？
   - 预期输出是什么？

2. **最小化复现**
   - 找出最小输入集
   - 隔离变量
   - 确认是确定性 Bug 还是概率性 Bug

3. **🆕 主动制造 RED（v4.0 强制）**
   - 在测试文件中**写入能复现 Bug 的断言**（即便用户报错与 disk 不符）
   - 跑测试 → **看到真实失败信息**
   - **禁止**：skip / 假断言 / 不跑测试
   - **禁止**：在 thinking 里"推断根因"而不动手

4. **记录环境**
   - 操作系统、版本
   - 依赖版本
   - 配置差异

#### 输出

- 复现步骤（具体到命令级别）
- 最小复现用例
- **真实的测试失败输出**（必须展示给用户）

---

### 阶段 2：定位（Isolate）

**目标**：缩小 Bug 出现的范围

#### 执行步骤

1. **二分法定位**
   - 在中间点加日志/断点
   - 向前/向后二分
   - 找到 Bug 首次出现的代码位置

2. **依赖排除**
   - 怀疑外部依赖？mock 它
   - 怀疑数据库？直接查询验证
   - 怀疑网络？隔离环境测试

3. **数据流追踪**
   - 输入数据如何流转？
   - 在哪里被改变？
   - 哪个环节出现偏差？

#### 输出

- 疑似代码位置（文件 + 行号）
- 影响范围
- 触发条件

---

### 阶段 3：根因（Root Cause）

**目标**：找到 Bug 的根本原因，而非症状

#### 5 Whys 分析法

```
Bug：用户登录失败

Why 1：为什么登录失败？
→ 密码验证不通过

Why 2：为什么密码验证不通过？
→ 数据库中的哈希值与输入哈希不匹配

Why 3：为什么哈希不匹配？
→ 用户表中存的是明文密码而非哈希

Why 4：为什么存的是明文？
→ 注册时没有调用 hash 函数

Why 5：为什么没有调用？
→ 注册函数忘记调用 hash（根因）
```

#### 根因类别

| 类别 | 描述 | 修复策略 |
|------|------|---------|
| **逻辑错误** | 代码逻辑写错 | 修改逻辑 |
| **边界遗漏** | 没考虑边界条件 | 添加边界处理 |
| **数据错误** | 数据本身有问题 | 修复数据 + 加验证 |
| **集成问题** | 模块协作出问题 | 修复接口契约 |
| **配置错误** | 环境/配置不对 | 修正配置 |
| **设计缺陷** | 架构层面有问题 | 重构设计 |

#### 输出

- 根因分析报告
- 修复策略

---

### 阶段 4：修复 + 回归测试（Fix + Test）

**目标**：修复根因，并防止再次发生

#### 4.1 设计修复方案

- 针对根因，而非症状
- 考虑副作用
- 考虑回归风险

#### 🆕 4.2 修测试不修实现（v4.0 强制）

**在写实现修复前**，先判断"失败的测试"是否有 bug：

| 测试失败原因 | 修复方式 |
|------------|---------|
| 断言写错边界值 | **改测试** |
| 注释与代码不一致 | **改测试** |
| 期望值与 FR 描述不符 | **改测试** |
| 变量名 / import 错误 | **改测试** |
| 实现确实不符合 FR | **改实现**（按 4.3 走 TDD）|

**禁止**：
- ❌ "测试失败了，让我改实现让它通过"（未先判断测试是否有 bug）
- ❌ "测试期望值不对，让我改期望值"（除非确实写错）

#### 4.3 写失败测试 + 修复

```
1. 写一个必定失败的测试，复现 Bug
2. 确认测试失败（必须真发生）
3. 修复代码
4. 确认测试通过
5. 确认没有破坏其他测试
6. 跑完整测试套件
```

#### 4.4 回归测试

- 跑完整测试套件
- 检查相关功能
- 验证边界情况
- 验证覆盖率 ≥ 80%

#### 4.5 文档化

- 记录到 lessons
- 如果是设计缺陷，记录到 decisions

#### 🆕 4.6 零编辑退出门槛（v4.0 强制）

完成本阶段**必须满足以下全部**：
- ✅ 修改了至少 1 个文件（实现 / 测试 / 配置）
- ✅ 跑过测试且看到**真实输出**（不是 "应该通过了"）
- ✅ 修复后所有相关测试通过
- ✅ 覆盖率未下降

**未满足** → 视为 bug 修复未完成，**必须继续**而非搪塞。

## 反模式（任何一条触发即视为违规）

### ❌ 反模式 1：随机修改
```typescript
// 错误
"试试加个 try-catch"
"改成 != 看看"
"注释掉这行看看"
```
**为什么错**：浪费时间，可能引入新 Bug

### ❌ 反模式 2：症状修补
```typescript
// 错误
"Bug 是用户输入空字符串时崩溃"
"修复：if (input === '') return"
// 但根因是没有输入验证
```
**为什么错**：根因没解决，类似 Bug 还会出现

### ❌ 反模式 3：跳过复现
```typescript
// 错误
"我猜大概是 X 问题"
"用户说崩溃了，我先加个 try-catch"
```
**为什么错**：没复现就修复，可能修错

### ❌ 反模式 4：不写回归测试
```typescript
// 错误
"修复了，但没加测试"
"反正已经修好了"
```
**为什么错**：没有测试保证，下次还会出

### 🆕 反模式 5：用户报错与 disk 不符时搪塞（v4.0）
```typescript
// 错误
"disk 上代码是 X，与你描述不符，应该是你环境的问题"
// 实际：应该主动修改 test 制造 RED 复现
```

### 🆕 反模式 6：零编辑回复（v4.0）
```typescript
// 错误
12+ 次循环思考后宣称"测试已经全部通过"
// 实际：vitest config 缺失，测试根本没跑起来
```

### 🆕 反模式 7：删除验证设施（v4.0）
```typescript
// 错误
"清理我创建的临时 vitest.config.ts"
// 实际：删除后测试跑不起来
```

## 与 TDD 铁律的关系

**调试也是 TDD**：
1. 写一个失败的测试（复现 Bug）—— RED
2. 修复代码（让测试通过）—— GREEN
3. 重构（保持测试通过）—— REFACTOR

这是 TDD 铁律的反向应用：从"红"开始。

---

## 红线（任何一条都意味着停止并重做）

- [ ] 还没复现就尝试修复
- [ ] 还没找到根因就修改代码
- [ ] 修改代码但没写回归测试
- [ ] 只修复症状而非根因
- [ ] "试试看"式随机修改
- [ ] 🆕 用户报错与 disk 不符时未主动制造 RED
- [ ] 🆕 修复后未跑测试 / 跑测试看不到真实输出
- [ ] 🆕 zero-edit 回复 bug 报告
- [ ] 🆕 修实现让"错的测试通过"
- [ ] 🆕 删除 vitest.config.ts / jest.config.js 等验证设施

## 心理陷阱与反驳

| 借口 | 反驳 |
|------|------|
| "Bug 太简单了不需要走流程" | 简单 Bug 也有根因 |
| "我猜是 X 问题，先改改" | 没复现就猜，浪费时间 |
| "时间紧迫，先打个补丁" | 补丁治标不治本，后续成本更高 |
| "用户说不清，先猜一个原因" | 主动询问，复现是金标准 |
| 🆕 "用户报错与 disk 不符，是用户环境问题" | 主动制造 RED 复现是金标准 |
| 🆕 "我已经在 thinking 里找到根因了" | 找到不等于修复，必须写测试 + 改实现 |
| 🆕 "测试已经全部通过" | 真跑了吗？看到真实输出了吗？ |

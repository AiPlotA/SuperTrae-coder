# Plan Devex Review 技能（开发者体验 4 维度 review）

> 基于 gstack `/plan-devex-review` Skill 改造
> 目的：让"5 分钟跑通 hello world"成为可验证的设计目标
> 强制级别：P1 - 任何 lib / API / CLI / 框架必走

---

## 4 维度 0-10 评分模板

### 维度 1：TTHW（Time To Hello World）—— 新用户多久能跑通第一个 happy path？

| 分 | 长什么样 |
|----|---------|
| 0 | 无文档 / 无 quick start |
| 3 | 有 README 但步骤 ≥10 步 |
| 5 | 5 步以内能跑通 happy path |
| 7 | 3 步以内 + 1 个命令 / 1 行代码 |
| **10** | **`npm install xxx && xxx.init` 30 秒内跑通,且过程中 0 决策点** |

### 维度 2：神奇时刻（Magical Moment）—— 用户用上后哪个瞬间会"哇"？

| 分 | 长什么样 |
|----|---------|
| 0 | 没有神奇时刻（只是个工具）|
| 3 | 1 个小神奇点（如自动补全）|
| 5 | 1 个明显神奇点（如一键部署）|
| 7 | 多个神奇点串联,首次使用 3 个"哇"瞬间 |
| **10** | **新用户 5 分钟内有 3 个"哇"瞬间,并主动分享给同事** |

### 维度 3：摩擦点（Friction Points）—— 用户卡在哪里？

| 分 | 长什么样 |
|----|---------|
| 0 | 摩擦点 ≥5 个,每个 ≥2 分钟 |
| 3 | 摩擦点 ≤3 个 |
| 5 | 每个摩擦点有"为什么这样设计"理由 |
| 7 | 摩擦点 ≤1 个,且是必要的 |
| **10** | **任意 1 个新用户 5 分钟内能完成核心任务,无任何"为什么这样"疑问** |

### 维度 4：用户画像追踪（Persona Trace）—— 3 个典型用户能跑通吗？

| 分 | 长什么样 |
|----|---------|
| 0 | 没考虑不同用户差异 |
| 3 | 1 个用户画像（自己）|
| 5 | 3 个用户画像（小白 / 中级 / 专家）|
| 7 | 3 个画像 + 各自的摩擦点清单 |
| **10** | **3 个画像实测 5 分钟,任何 1 个跑不通就回去修** |

## 3 强制问题

### 问题 1：新用户跑通 hello world 的最短路径是什么？

> 不要接受"看 README"。追问实测：
> - 计时：1 个新用户从 clone 到跑通多久？
> - 决策点：过程中需要做几个选择？每个选择有默认值吗？
> - 错误处理：任何 1 步失败,有清晰的"做什么能恢复"吗？

### 问题 2：用户第 1 次用的"神奇时刻"是哪个瞬间？

> 不要接受"用起来很快"。追问到具体瞬间：
> - 是看到结果？（例：1 键部署完）
> - 是省了事？（例：自动生成 boilerplate）
> - 是解决了痛点？（例：1 步迁移旧数据）
> - 怎么让这个瞬间更早出现 / 更明显？

### 问题 3：3 个真实用户画像实测过吗？

> 不要接受"我们测试过"。追问到具体人：
> - 小白（第一次用）：卡在哪？
> - 中级（用过类似工具）：哪些 API 命名不一致？
> - 专家（用过同类最佳）：哪些高级功能缺失？
> - 找 3 个真人实测,记录卡点

## 反模式

- ❌ "我们的文档很全"（文档 ≠ 易用性,实测计时才算）
- ❌ quick start 用最新功能做示例（新用户配环境就放弃了）
- ❌ 摩擦点"是用户的问题"（设计师的失败,不是用户的）
- ❌ 用户画像是"理想用户"而非真实用户（理想用户不存在）

# 哲学审计（Philosophy Audit）

> 基于工程哲学 v1 的 9 条款宪法
> 目的：审查代码/配置/项目是否遵守本项目的工程哲学

---

## 触发场景

- 用户说"审计"、"审查"、"哲学合规"、"9 条款"
- 提交 PR 前
- 完成重大功能后
- 月度/季度项目审查

---

## 9 条款审查清单

### 条款 I：规范先行

> 每个功能开发必须先有 Spec

- [ ] 提交的功能有对应 spec.md
- [ ] spec 包含：用户场景 + FR-XXX + 关键实体 + 成功标准
- [ ] spec 中 `[NEEDS CLARIFICATION]` ≤ 3 个
- [ ] 变更使用 Delta Spec 机制

**评分标准**：
- 0 分：无 spec
- 5 分：有 spec 但缺关键章节
- 10 分：完整 spec + Delta Spec 流程

### 条款 II：TDD 铁律

> NO PRODUCTION CODE WITHOUT A FAILING TEST FIRST

- [ ] 测试覆盖率 ≥ 80%
- [ ] 测试先于实现（git log 中测试 commit 早于实现）
- [ ] 无 mock 滥用
- [ ] 无私有方法测试
- [ ] 测试有 AAA 模式

**评分标准**：
- 0 分：无测试
- 5 分：有测试但覆盖率 < 80%
- 10 分：覆盖率 ≥ 80% + 严格 TDD

### 条款 III：库优先

> 每个功能作为独立模块实现

- [ ] 功能代码在 `lib/` 或 `packages/`
- [ ] 每个模块有独立测试
- [ ] 模块有清晰公共 API
- [ ] 模块间通过接口通信
- [ ] 无全局变量

**评分标准**：
- 0 分：单文件堆叠
- 5 分：基本模块化
- 10 分：完整库优先设计

### 条款 IV：简洁性（YAGNI）

> 每个功能实现不超过 3 个新文件

- [ ] 每个功能 ≤ 3 个新文件
- [ ] 无提前抽象
- [ ] 无 3 个相似实现前的抽象
- [ ] 优先简单方案

**评分标准**：
- 0 分：严重过度工程
- 5 分：基本简洁
- 10 分：极致简洁

### 条款 V：可观测性

> 所有关键操作必须可观测

- [ ] 使用结构化日志（JSON）
- [ ] 每个 API 有错误处理
- [ ] 关键路径记录执行时间
- [ ] 敏感操作有审计日志
- [ ] 禁止 `console.log` 字符串拼接

**评分标准**：
- 0 分：无日志
- 5 分：基本日志
- 10 分：完整结构化日志 + 审计

### 条款 VI：安全边界

> 禁止危险操作

- [ ] 无 `rm -rf`（/tmp 除外）
- [ ] 无 `git push --force`
- [ ] 无数据库 `DROP`
- [ ] 无明文环境变量
- [ ] 外部 API 调用有审计

**评分标准**：
- 0 分：危险操作
- 5 分：基本合规
- 10 分：完全合规

### 条款 VII：完整实现

> 每个任务做透

- [ ] 无 TODO/FIXME（除非有追踪 issue）
- [ ] 包含错误处理
- [ ] 包含边界情况
- [ ] 包含测试
- [ ] 文档已更新

**评分标准**：
- 0 分：大量 TODO
- 5 分：基本完整
- 10 分：100% 完整

### 条款 VIII：证据优于断言

> 跑命令看输出再宣称成功

- [ ] "完成"前有命令输出
- [ ] 测试结果明确
- [ ] 失败明确报告
- [ ] 无"应该没问题"

**评分标准**：
- 0 分：纯断言
- 5 分：偶尔跑命令
- 10 分：100% 证据驱动

### 条款 IX：用户主权

> AI 推荐，用户决定

- [ ] 重大决策有 AskUserQuestion
- [ ] 推荐方案时说明利弊
- [ ] 不确定时主动暂停
- [ ] 不跳过 verification

**评分标准**：
- 0 分：完全自主
- 5 分：偶尔询问
- 10 分：100% 用户主权

## 审计报告模板

```markdown
# 哲学合规性审计报告

## 基本信息
- 审查对象：[文件/项目/功能]
- 审查时间：YYYY-MM-DD
- 审查者：philosophy-audit skill

## 9 条款评分

| 条款 | 评分 | 评级 | 关键问题 |
|------|------|------|---------|
| I 规范先行 | 8/10 | ✅ | 缺 Delta Spec |
| II TDD 铁律 | 7/10 | ⚠️ | 覆盖率 78% < 80% |
| III 库优先 | 9/10 | ✅ | 全局变量 1 个 |
| IV 简洁性 | 8/10 | ✅ | 1 个文件 > 500 行 |
| V 可观测性 | 5/10 | ⚠️ | console.log 滥用 |
| VI 安全边界 | 10/10 | ✅ | |
| VII 完整实现 | 7/10 | ⚠️ | 2 个 TODO |
| VIII 证据优于断言 | 9/10 | ✅ | |
| IX 用户主权 | 8/10 | ✅ | |
| **综合** | **7.9/10** | ✅ | |

## 关键发现

### P0（必须立即修改）
- 条款 V：所有 console.log 改为结构化日志
- 条款 VII：清理 2 个 TODO（添加 issue 链接）

### P1（应该修改）
- 条款 II：覆盖率提升至 80%
- 条款 I：补充 Delta Spec 模板

### P2（可选改进）
- 条款 IV：拆分 500 行文件

## 改进建议

### 短期（1 周）
1. 用 logger 替换所有 console.log
2. 补充缺失测试（覆盖率 78% → 85%）

### 中期（1 月）
1. 建立 Delta Spec 流程
2. 重构全局变量

## 结论
- [ ] 通过：综合 ≥ 7.0
- [ ] 修订：综合 5.0-7.0
- [ ] 否决：综合 < 5.0
```
